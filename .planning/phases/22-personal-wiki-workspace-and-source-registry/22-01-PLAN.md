<data_flow>
1. User runs `wikify init [BASE]`.
2. Wikify creates visible product directories and internal control directories under the workspace root.
3. Wikify writes `wikify.json` with schema `wikify.workspace.v1`, a stable `workspace_id`, and workspace-relative path settings.
4. Wikify writes `.wikify/registry/sources.json` with schema `wikify.source-registry.v1`, the same `workspace_id`, `updated_at`, and an empty `sources` object.
5. User runs `wikify source add <locator> --type file|directory|url|repository|note`.
6. Wikify loads the workspace manifest and registry, normalizes a `locator_key`, performs only bounded offline metadata probing, and writes a source record if the locator is new.
7. Re-adding the same `locator_key` returns the existing source without creating a duplicate.
8. User or agent runs `wikify source list` or `wikify source show <source_id|locator>` to receive stable JSON envelope output.
</data_flow>

<threat_model>
## Threat Model

**Assets:**
- Workspace manifest path settings.
- Source registry integrity.
- Local filesystem paths recorded as source locators.
- Agent-facing JSON outputs consumed by future automation.

**Threats and mitigations:**
- Path traversal or unintended writes: write only inside the selected workspace root for `wikify.json`, `.wikify/registry/sources.json`, `sources/`, `wiki/`, `artifacts/`, and `views/`.
- Hidden network activity: URL and remote repository registration must not fetch, ping, clone, or resolve remote resources in Phase 22.
- Registry corruption: write JSON through a temporary file followed by atomic replace.
- Identity instability: use opaque generated `source_id`; never derive source identity from mutable locator or content fields.
- Duplicate registry bloat: build a runtime `locator_key` index and make duplicate add idempotent.
</threat_model>
