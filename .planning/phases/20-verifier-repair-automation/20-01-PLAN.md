<objective>
Build verifier repair automation without adding hidden provider behavior.

Purpose: after a verifier blocks a task, an explicit producer agent should be able to receive the rejection feedback, regenerate the rejected bundle, and pass through the same verifier/apply gates without asking the user to manually delete stale artifacts.
</objective>

<data_flow>
1. A prior verifier rejection leaves task status `blocked`, `blocked_feedback`, verification artifact, rejected bundle, and block event details.
2. A caller runs `run-task --id <id> --agent-command ... --verifier-command ...` or batch selects blocked tasks with explicit commands.
3. The runner recognizes verifier-blocked work, applies lifecycle retry/mark-proposed, and treats the existing rejected default bundle as stale.
4. The runner writes a fresh bundle request with `repair_context` from task feedback or latest block event.
5. The producer receives that request and overwrites the suggested bundle path.
6. The verifier reviews the new bundle.
7. Accepted bundles apply and mark done; rejected bundles leave content unchanged and persist fresh blocked feedback.
</data_flow>
