# Phase 28: Maintenance Integration And Compatibility - Research

**Researched:** 2026-04-30
**Question:** What must be true to plan Phase 28 well?

## Research Summary

Phase 28 should extend the existing maintenance system, not replace it. The current v0.1.0a2 maintenance loop is already valuable: it builds graph artifacts, derives graph findings, plans deterministic/queued actions, creates agent task queues, scopes patch proposals, hands off bundle requests to explicit agents, verifies bundles through explicit verifier commands, applies deterministic `replace_text` patches, records rollback data, and can run batches/loops.

The gap is identity and preservation. Today maintenance mostly thinks in graph subjects and relative paths. v0.2.0 added canonical wiki objects, generated page metadata, human view manifests, agent exports, context packs, validation records, and generated-page source traceability. Phase 28 must make maintenance resolve those artifacts without changing the legacy command surface.

## Existing Maintenance Pipeline

Current data flow:

1. `wikify.maintenance.runner.run_maintenance()` calls `build_graph_artifacts(root, include_html=False)`.
2. `wikify.maintenance.findings.build_findings(graph)` derives graph-only findings from `graph.analytics`.
3. `wikify.maintenance.planner.build_plan(findings, policy)` creates `wikify.maintenance-plan.v1`.
4. `wikify.maintenance.executor.apply_plan(plan)` marks deterministic actions executed and semantic/generated-content actions queued.
5. `wikify.maintenance.task_queue.build_task_queue(plan, execution, findings)` writes `wikify.graph-agent-tasks.v1` under `sorted/graph-agent-tasks.json`.
6. `propose`, `bundle-request`, `produce-bundle`, `verify-bundle`, `apply`, `rollback`, `run-task`, `run-tasks`, `maintain-run`, and `maintain-loop` compose those artifacts.

Compatibility implication: keep `sorted/graph-*` artifacts and `wikify.graph-agent-tasks.v1` stable. Add optional fields only.

## v0.2 Artifact Inputs

Useful artifact sources:

- `artifacts/objects/**/*.json` and `artifacts/objects/object-index.json` from `wikify.objects`.
- `wiki/pages/*.md` generated by `wikify wikiize`, with front matter carrying `id`, `body_path`, `source_refs`, `review_status`, and related metadata.
- `.wikify/sync/source-items.json` from incremental sync.
- `.wikify/queues/wikiization-tasks.json` for generated page drift and source-backed wikiization issues.
- `.wikify/views/view-manifest.json`, `.wikify/views/last-views.json`, and `.wikify/queues/view-tasks.json` from human view generation.
- `llms.txt`, `llms-full.txt`, `artifacts/agent/*.json`, `.wikify/agent/last-agent-export.json`, and `.wikify/agent/context-pack-manifest.json` from agent exports.
- `artifacts/objects/validation.json` from object validation.
- Existing `graph/graph.json` and graph analytics.

Important boundary: maintenance should not reread raw source files or silently run `sync`, `wikiize`, `views`, `agent export`, external agents, providers, embeddings, or vector stores.

## Recommended Architecture

Add a small maintenance target resolver, likely `wikify/maintenance/targets.py`, with a schema such as `wikify.maintenance-targets.v1`.

It should:

- Load optional v0.2 artifacts defensively.
- Index targets by object id, body path, object path, view path, agent artifact path, source id, and graph subject.
- Return normalized target dictionaries with fields such as `target_kind`, `target_family`, `object_id`, `object_type`, `body_path`, `object_path`, `view_path`, `agent_artifact_path`, `source_refs`, `review_status`, `write_scope`, `regeneration_command`, and `warnings`.
- Degrade gracefully when object/view/agent artifacts are absent so legacy workspaces still work.

Then compose this resolver into findings and task queue building:

- Existing graph findings remain unchanged when no target resolver metadata exists.
- When a graph subject maps to a generated page or object, enrich the finding and queued task with optional typed metadata.
- Add v0.2 artifact-health findings for validation records, wikiization/view tasks, missing or stale views, and missing agent exports.
- Preserve legacy task fields (`target`, `evidence`, `write_scope`, `action`, etc.).

## Preservation Research

Generated wiki pages have two metadata carriers:

- Markdown front matter in `wiki/pages/*.md`.
- Canonical JSON object under `artifacts/objects/wiki_pages/<page-id>.json`.

Patch application can currently modify any scoped file with exact-once `replace_text`; it does not understand generated-page metadata. Verifier acceptance is delegated to an explicit verifier command, which is useful but not sufficient. Phase 28 needs deterministic local checks that reject bundles removing or silently changing `source_refs` and `review_status`.

Recommended helper: `wikify/maintenance/preservation.py`.

Useful functions:

- `build_preservation_context(base, proposal)` returns generated-page metadata for proposal write scope.
- `validate_preservation_before_apply(base, proposal, bundle)` simulates patch output and rejects changes that alter `source_refs` or `review_status`.
- `attach_preservation_context_to_request(base, proposal)` gives external agents/verifiers explicit facts to preserve.

Where to integrate:

- `proposal.py`: include preservation metadata for generated page tasks.
- `bundle_request.py`: add agent instructions and safety block for generated page preservation.
- `bundle_verifier.py`: include preservation facts in the verifier request and run deterministic preservation validation before accepting.
- `patch_apply.py`: run the same local preservation validation before apply so unsafe bundles cannot bypass verifier.
- `task_runner.py`: preserve existing verifier rejection feedback path.

## Validation Architecture

Automated validation should cover:

- Unit tests for target resolver loading, indexing, and legacy degradation.
- Unit tests for graph finding enrichment and optional task metadata propagation.
- Unit tests for validation/view/agent artifact findings and regeneration tasks.
- Unit tests for generated-page preservation rejection in bundle request, verifier, and apply flows.
- CLI/integration tests proving v0.1.0a2 maintenance commands remain compatible.
- A local E2E fixture covering `init -> source add -> sync -> wikiize -> views -> agent export/context -> maintain -> task/proposal/verifier/repair`.

Recommended commands:

```bash
python3 -m unittest tests.test_maintenance_targets tests.test_maintenance_findings tests.test_maintenance_task_queue -v
python3 -m unittest tests.test_maintenance_artifact_findings -v
python3 -m unittest tests.test_maintenance_generated_page_preservation -v
python3 -m unittest tests.test_maintenance_e2e -v
python3 -m unittest discover -s tests -v
python3 -m compileall -q wikify
```

## Risks

- Object-aware code could accidentally break legacy Markdown-only workspaces. Countermeasure: compatibility tests with `sample-kb` and task queue schema assertions.
- Direct patching of derived views or agent exports could create stale artifacts. Countermeasure: regeneration tasks with explicit commands rather than manual patches.
- Generated page front matter and object JSON can drift. Countermeasure: preservation checks plus focused object validation after apply.
- Planner/executor may try to overfit everything into `findings.py`. Countermeasure: isolate target resolution and preservation helpers.

## RESEARCH COMPLETE

Plan Phase 28 as four waves:

1. Object-aware target resolver and graph task enrichment.
2. Artifact-health findings for validation, view, wikiization, and agent exports.
3. Generated-page preservation guard across proposal, bundle, verifier, and apply.
4. Compatibility documentation, E2E verification, and GSD completion artifacts.
