# Plan 28-03: Enforce Generated Page Source-Ref And Review-Status Preservation

<objective>
Add local preservation checks so repair and verifier flows cannot remove or silently alter generated wiki page `source_refs` or `review_status`, while keeping the existing patch proposal, bundle request, verifier, apply, rollback, and repair feedback contracts compatible.
</objective>

<data_flow>
1. A generated-page task enters `build_patch_proposal()`.
2. The proposal records normal `write_scope` plus a `preservation` block for each generated wiki page in scope.
3. `build_bundle_request()` includes the preservation block in request JSON and adds explicit agent instructions to preserve `source_refs` and `review_status`.
4. Patch bundle preflight simulates `replace_text` operations.
5. A preservation validator compares before/after front matter and object JSON metadata for generated wiki pages.
6. `verify_patch_bundle()` and `apply_patch_bundle()` both reject unsafe preservation changes with structured errors before content is applied.
7. Rejected verifier output continues to block the task using the existing lifecycle feedback path.
</data_flow>

<threat_model>
## Threat Model

**Assets:** generated wiki page front matter, wiki page object JSON, source refs, review status, patch bundles, verifier verdicts, rollback records.

**Threats and mitigations:**
- Metadata loss by external agent: bundle request includes preservation facts and local preflight rejects unsafe output.
- Weak verifier: `apply_patch_bundle()` performs local preservation validation even when no external verifier is used.
- Front matter/object JSON divergence: preservation context includes both carriers and rejects silent mismatch when either is touched.
- Repair-loop bypass: verifier rejection still uses existing blocked feedback and repair context.
- Patch model creep: no new operations; validation simulates existing `replace_text` only.
</threat_model>
