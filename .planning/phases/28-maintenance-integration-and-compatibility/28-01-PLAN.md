# Plan 28-01: Build Object-Aware Maintenance Targets And Task Enrichment

<objective>
Add a v0.2 maintenance target resolver and wire it into existing graph findings and task queue generation so `wikify maintain` can read personal wiki object metadata while preserving legacy maintenance behavior.

<data_flow>
1. `run_maintenance(base, policy, dry_run)` resolves the workspace root.
2. It builds graph artifacts exactly as before.
3. It calls a new target resolver such as `load_maintenance_targets(root)`.
4. The resolver reads existing object JSON, object index, source item index, validation report, view manifest/report, view task queue, agent export artifacts, and graph artifact paths when present.
5. `build_findings(graph, targets=...)` keeps existing graph findings but enriches subjects with v0.2 target metadata when a subject maps to an object id, body path, object path, source view path, or agent artifact path.
6. `build_task_queue(plan, execution, findings)` copies optional typed metadata from findings into queued tasks while preserving legacy task fields.
7. Legacy workspaces without object artifacts still produce the same graph findings, plan, and task queue shape.
</data_flow>

<threat_model>
## Threat Model

**Assets:** legacy maintenance artifacts, v0.2 object metadata, task queue schema, graph artifacts, source refs, review status.

**Threats and mitigations:**
- Legacy breakage: keep existing fields and schema version; add tests over `sample-kb` and legacy graph subjects.
- Second source of truth: resolver is ephemeral and derived from existing artifacts.
- Malformed optional artifacts: record warnings and continue where safe; structured errors only for required malformed reads.
- Overbroad task writes: keep path-based `write_scope` relative to workspace root.
- Hidden execution: target loading must not run sync/wikiize/views/agent export/provider commands.
</threat_model>
